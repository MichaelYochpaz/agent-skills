# GitHub Actions

Viewing workflow runs, debugging failures, rerunning jobs, and managing workflows with `gh run` and `gh workflow`. Mutation commands (`rerun`, `cancel`, `delete`, `workflow run`, `workflow enable/disable`) modify GitHub state — see Safety in the main skill body.

## Commands

### List runs

List recent workflow runs in a repository, filtered by workflow, branch, status, user, or event type.

```bash
gh run list --repo owner/repo
gh run list --repo owner/repo --workflow ci.yml --branch main --status failure --limit 10
```

Flags:
- `--workflow NAME` -- Filter by workflow file or name
- `--branch NAME` -- Filter by branch
- `--status VALUE` -- Filter by status (see values below)
- `--user USERNAME` -- Filter by triggering user
- `--event NAME` -- Filter by event type (e.g., `pull_request`, `push`)
- `--commit SHA` -- Filter by commit SHA
- `--created RANGE` -- Filter by creation date (e.g., `">2024-01-01"`)

JSON fields: `attempt`, `conclusion`, `createdAt`, `databaseId`, `displayTitle`, `event`, `headBranch`, `headSha`, `name`, `number`, `startedAt`, `status`, `updatedAt`, `url`, `workflowDatabaseId`, `workflowName`

Status values: `queued`, `completed`, `in_progress`, `requested`, `waiting`, `pending`, `action_required`, `cancelled`, `failure`, `neutral`, `skipped`, `stale`, `startup_failure`, `success`, `timed_out`

### View a run

View a workflow run's summary, job statuses, step details, and logs. Start with `--log-failed` when debugging — `--log` can produce very large output.

```bash
gh run view <run-id>                     # Summary with jobs
gh run view <run-id> -v                  # Verbose (shows individual steps)
gh run view <run-id> --log-failed        # Logs for failed steps only (start here)
gh run view <run-id> --job <job-id> --log  # Full log for a specific job
```

Flags:
- `-v, --verbose` -- Show individual job steps and their status
- `--log` -- Full log output (can be very large)
- `--log-failed` -- Logs for failed steps only
- `--job JOB-ID` -- Scope to a specific job within the run
- `--attempt N` -- View a specific attempt
- `--exit-status` -- Exit non-zero if run failed (useful for scripting)

JSON fields: `attempt`, `conclusion`, `createdAt`, `databaseId`, `displayTitle`, `event`, `headBranch`, `headSha`, `jobs`, `name`, `number`, `startedAt`, `status`, `updatedAt`, `url`, `workflowDatabaseId`, `workflowName`

Log caveats:
- On `gh` versions before 2.92, `gh run view --log` / `--log-failed` did not sanitize terminal escape sequences, allowing malicious logs to manipulate the terminal. Verify the installed version if log output looks corrupted or suspicious.
- `gh run view` may fall back to slower per-job log fetching when platform log archives cannot be matched to jobs. If more than 25 job logs are missing, the operation can fail.
- Some log lines may appear under `UNKNOWN STEP` when GitHub cannot associate them with a workflow step.

### Rerun

Rerun an entire workflow run, only failed jobs, or a specific job. Useful after pushing a fix or for transient failures.

```bash
gh run rerun <run-id>                    # Rerun all jobs
gh run rerun <run-id> --failed           # Rerun only failed jobs
gh run rerun <run-id> --failed --debug   # Rerun with debug logging enabled
```

Flags:
- `--failed` -- Rerun only failed jobs
- `--job JOB-ID` -- Rerun a specific job
- `--debug` -- Enable debug logging for the rerun

Note: `--job` requires the job's `databaseId`, not the number from the URL. Get it with:
```bash
gh run view <run-id> --json jobs --jq '.jobs[] | {name, databaseId}'
```

### Cancel / Delete

Cancel an in-progress workflow run, or delete a completed run from the history.

```bash
gh run cancel <run-id>
gh run delete <run-id>
```

### Download artifacts

Download artifacts generated by a workflow run. Each artifact is extracted into a separate directory named after the artifact.

```bash
gh run download <run-id>
```

Flags:
- `--name NAME` -- Download a specific artifact by name
- `--dir PATH` -- Download to a specific directory

### Watch

Watch a workflow run in real-time until it completes, showing progress updates.

```bash
gh run watch <run-id>
gh run watch <run-id> --interval 5       # Refresh every 5 seconds
```

### PR-associated checks

Show CI/CD check statuses for a specific pull request. Use this instead of `gh run list` when you want to see checks tied to a PR rather than workflow runs by branch.

```bash
gh pr checks 123
gh pr checks 123 --json name,state,bucket,link --jq '.[] | select(.bucket == "fail")'
gh pr checks 123 --watch
```

### Workflows

List, view, enable, and disable GitHub Actions workflows in a repository.

```bash
gh workflow list                         # List all workflows
gh workflow view ci.yml                  # View workflow info
gh workflow view ci.yml --yaml           # View workflow YAML definition
gh workflow enable ci.yml
gh workflow disable ci.yml
```

### Trigger a workflow manually

Manually trigger a workflow run by creating a `workflow_dispatch` event. The workflow must have an `on: workflow_dispatch` trigger defined. Input parameters can be passed as key-value pairs.

```bash
gh workflow run ci.yml
gh workflow run deploy.yml --ref develop -f environment=staging -f version=1.2.0
```

Flags:
- `--ref BRANCH` -- Trigger on a specific branch/tag
- `-f key=value` -- String input parameter
- `-F key=value` -- Typed parameter (booleans/integers auto-convert; `@file` reads from file)
- `--json` -- Read input parameters as JSON from stdin

### Other Commands

These commands are rarely needed. Run `gh <command> --help` for flags and usage.

- `gh cache list` / `gh cache delete` -- List and delete GitHub Actions caches for a repository. Useful for debugging cache-related build issues.

## Workflows

### Debugging a failed Action

1. **Find the failed run:**
   ```bash
   gh run list --repo owner/repo --status failure --limit 5
   ```

2. **View the run summary with job steps:**
   ```bash
   gh run view <run-id> -v
   ```

3. **Read the logs for failed steps:**
   ```bash
   gh run view <run-id> --log-failed
   ```

4. **If more detail is needed, read the full log for a specific job:**
   ```bash
   gh run view <run-id> --job <job-id> --log
   ```

5. **View the workflow definition to understand the pipeline:**
   ```bash
   gh workflow view <workflow-name> --yaml
   ```

   Or fetch it via API if you need a specific branch version:
   ```bash
   gh api "repos/OWNER/REPO/contents/.github/workflows/ci.yml?ref=<branch>" -H "Accept: application/vnd.github.raw+json"
   ```

6. **After fixing, rerun failed jobs:**
   ```bash
   gh run rerun <run-id> --failed
   ```

### Investigating a PR's CI failures

1. **Check which checks failed:**
   ```bash
   gh pr checks 123 --json name,state,bucket,link --jq '.[] | select(.bucket == "fail")'
   ```

2. **Find the corresponding run:**
   ```bash
   gh run list --branch <pr-head-branch> --limit 5
   ```

3. **Read the failed logs:**
   ```bash
   gh run view <run-id> --log-failed
   ```

4. **Rerun after pushing a fix:**
   ```bash
   gh run rerun <run-id> --failed
   ```
